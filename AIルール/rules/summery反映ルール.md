プロジェクトの要約プロンプト

あなたは、このプロジェクトの仕様と構成を整理するテクニカルライター兼ソフトウェアアーキテクトです。
これから、プロジェクトに関するドキュメントやソースコードの一部を渡します。

## あなたのゴール

* 今後の実装タスク（複数 issue）をこなすときに参照できる、**簡潔で再利用しやすい要約**を作ってください。
* 要約は **「仕様」「ルール」「構成」** の3つを中心に整理します。

##入力
- 設計書は`docs`配下に格納しています。
- 作成プログラムは`src`配下に格納しています。
これらを読み込み、「ゴール」「出力フォーマット」「厳守事項」に従って要約してください。

##出力
**出力は、指定したセクション構成に従った Markdown 形式でお願いします。**
**ファイルはdocs配下に`Summery.md`として作成・更新お願いします **


## 出力フォーマット

以下のセクション構成で、日本語で要約してください。

1. プロジェクト概要

   * このプロジェクトが何をするものか
   * 主なユースケース／利用者イメージ

2. 機能仕様（重要ポイントのみ）

   * 中心となる機能の一覧
   * 仕様上重要な前提条件・ドメインルール
   * 入出力のフォーマットや制約（あれば）

3. 実装ルール・コーディング規約

   * 命名規則、ディレクトリ構成のルール
   * 使用が推奨／禁止されているライブラリ・書き方
   * テスト方針（どのレイヤーでテストを書くか など）

4. アーキテクチャ・構成

   * ディレクトリ構成と各ディレクトリの責務
   * 主要なクラス・モジュールと、その役割
   * 外部サービス／外部APIとの連携（あれば）

5. 重要な制約・注意点

   * パフォーマンス・セキュリティ・スレッドセーフティなどの制約
   * 既存実装に手を入れる際に特に注意すべきポイント

6. 不明点・あいまいな点

   * 入力からは判断しきれなかった点
   * ドキュメント間で矛盾している・解釈が分かれそうな点

## トーンと粒度

* 将来の自分や別の開発者が「この要約だけ読めば、このプロジェクトの文脈が思い出せる／把握できる」レベルを目指してください。
* 原文の写経ではなく、**情報を圧縮して再構造化**してください。
* 細かすぎる実装詳細（1つの関数のロジックなど）は、必要な場合のみ代表例として触れてください。

## 厳守してほしいこと

* 入力に書かれていないことは、**推測で断定しないでください。**

  * 推測せざるをえない場合は、その旨を明示し、「推測」「想定」とラベルを付けてください。
* どのファイルやドキュメントから読み取った情報なのかが重要な場合、

  * 可能であれば「例：from `{{ファイル名}}`」のように軽く触れてください（細かい行番号などは不要）。
* 入力が明らかに不足していて要約が粗くなりそうな場合は、

  * セクション 6「不明点・あいまいな点」で、**不足している情報の種類**をリストアップしてください。

## Summery.md 更新ルール

### 1. 役割と対象範囲

- `Summery.md` は「プロジェクトの骨格（概要・主要機能・構成・共通ルール）」を圧縮して示すための要約ファイルとする。
- 詳細仕様や関数レベルのロジックは、設計書（例: `docs/` 配下）やソースコード（例: `src/` 配下）が一次情報であり、`Summery.md` はそれらへのナビゲーション兼ダイジェストと位置づける。

### 2. 更新が必要になる変更

次のような変更があった場合にのみ、`Summery.md` の更新を検討する。

- 主要機能の追加・削除・役割変更
- ディレクトリ構成・レイヤ構成など、アーキテクチャに関わる変更
- 全体に効く共通ルール・前提条件の変更（例: 命名規則、使用禁止ライブラリ、テスト方針）
- 今後の ISSUE でも繰り返し参照されることが想定される設計判断・制約

上記に該当しない「単一 ISSUE に閉じた特殊な挙動・パラメータ変更」などは、原則として `Summery.md` に書かず、該当コードや設計書にのみ反映する。

### 3. 更新手順

1. 対応した ISSUE の差分と関連ドキュメント（`docs/` 配下）を確認し、上記「2. 更新が必要になる変更」に該当するかを判断する。
2. 該当する場合のみ `Summery.md` を開き、影響するセクション（概要 / 機能仕様 / 実装ルール / アーキテクチャ / 注意点 / 不明点など）を特定する。
3. 実際の設計書・コードを一次情報として参照しつつ、本ファイルに定義されている「プロジェクトの要約プロンプト」の方針に従って、該当部分だけを更新する。
4. 更新後、`Summery.md` の先頭付近に「最終更新日」と「どの Issue / バージョンまで反映済みか」を追記または更新する。

### 4. 記述ルール（品質基準）

- 事実と推測は必ず分離し、推測や暫定判断を記載する場合は「推測」「未確認」「暫定」などと明示する。
- 元の設計書・コードに存在しない情報を勝手に追加しない。あくまで情報の要約・再構造化の範囲にとどめる。
- 単一 Issue に閉じた細かい挙動・パラメータは原則 `Summery.md` に書かず、必要に応じて該当ドキュメントやコードへの参照（ファイルパスなど）を記載する。
- 今後の Issue で何度も参照される内容かどうかを基準に、不要な詳細は入れない。
- 日本語は簡潔・平易にし、箇条書きを優先して冗長な説明を避ける。

### 5. ISSUE 完了時のチェックリスト

各 ISSUE の実装・テスト完了時に、次を確認する。

- [ ] この Issue により、プロジェクトの概要・主要機能・構成・共通ルールに変化があるか？
- [ ] 変化がある場合、その内容は今後の Issue でも参照されるか？
- [ ] 参照される場合、本ルールに従って `Summery.md` を更新したか？
- [ ] `Summery.md` の最終更新日と反映済みの Issue / バージョン情報を更新したか？

`Summery.md` を新規に作成する場合や、大きく書き換える場合は、本セクションに加えて、このファイル先頭の「プロジェクトの要約プロンプト」を codex に渡し、生成された要約をベースに手動で微調整する。
